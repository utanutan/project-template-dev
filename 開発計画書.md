## 1. 開発方針 (Development Policy)

### 1.1 技術選定 (Technology Stack)
本プロジェクト（Life OS基盤構築）および、この基盤上で稼働するエージェント群の技術スタック。

*   **Core Platform**: Google Antigravity (IDE/Agentic Environment)
*   **Execution Engine**: OpenCode (CLI Tool for Agent Execution)
*   **Agent Models (Guild Members)**:
    *   *Implementation*: DeepSeek-V3
    *   *Architecture/Design*: Claude 3.5 Sonnet / Gemini 1.5 Pro
    *   *Creative/Content*: GPT-4o / Sakana AI
    *   *Legal/Compliance*: ELYZA (Latest)
*   **Infrastructure**: Local Environment (Mac OS)
*   **Configuration**: JSON (`agents.json`), Environment Variables (`.env`)

### 1.2 アーキテクチャ設計 (System Architecture)
複数のプロジェクト（開発、執筆、生活管理など）を並列で実行・管理するためのディレクトリ構成。汎用性と拡張性を最優先する。

#### ディレクトリ構造 (Folder Structure)
```
/Users/carpediem/workspace/AntiGravity/project-template-dev/  (Root)
├── config/                  # システム全体の設定
│   ├── agents.json          # エージェントの役割・性格・モデル定義
│   └── common_settings.env  # APIキーや共通定数
├── library/                 # 再利用可能なテンプレート・指示書
│   ├── dev-templates/       # 開発プロジェクト用 (Rust, Next.js等)
│   ├── creative-templates/  # コンテンツ制作用 (Blog, SNS等)
│   └── life-templates/      # 生活管理用 (Trip Plan, Research等)
├── inbox/                   # 全ての「種」が入る場所 (Raw Ideas)
│   └── (user_notes.md, random_thoughts.txt...)
└── projects/                # アクティブなプロジェクトの実体
    ├── dev-app-prototype/   # [Type: Dev] アプリ開発プロジェクト
    ├── blog-tech-entry/     # [Type: Creative] 記事執筆プロジェクト
    └── plan-vacation/       # [Type: Life] 旅行計画プロジェクト
```

#### 各ディレクトリの役割詳細
1.  **`config/`**: ギルドメンバー（エージェント）の名簿や性格付けを管理する“人事部”。ここを変更するだけで、プロジェクト全体のアサインを変更可能。
2.  **`library/`**: 過去の成功パターンや標準プロセスを蓄積する“図書館”。新しいプロジェクトを始める際は、ここから「型」をコピーして開始する。
3.  **`inbox/`**: ユーザーの脳内にある未整理のアイデアを投げ込む“ポスト”。エージェントがここを定期巡回し、プロジェクト化を提案する。
4.  **`projects/`**: 実際に動いている“作業現場”。各ディレクトリが独立したワークスペースとして機能する。

## 2. 実装計画 (Implementation Steps)

### Phase 1: Foundation (基盤構築)
*   [ ] **ディレクトリ構造の整備**: 上記アーキテクチャに基づくフォルダ作成。
*   [ ] **Configuration定義**: `agents.json` のスキーマ設計と初期メンバー（DeepSeek, Gemini, etc.）の登録。
*   [ ] **Environment設定**: 必要なAPIキー管理の仕組み（`.env` 等）の確立。

### Phase 2: Template System (テンプレート整備)
*   [ ] **開発計画書・プロジェクト計画書のテンプレート化**: どのプロジェクトでも使い回せるMarkdownテンプレートの作成 (`library/` 配下)。
*   [ ] **基本エージェント指示書**: 「Architect」「Developer」「Editor」それぞれの基本プロンプトの作成。

### Phase 3: Workflow Automation (自律動作検証)
*   [ ] **Scaffold Script**: `library/` のテンプレートを使って `projects/` に新規プロジェクトを立ち上げるスクリプト（またはWorkflow）の作成。
*   [ ] **Inbox Processing**: `inbox/` のメモから適切なテンプレートを選択し、プロジェクト化を提案するフローのテスト。

## 3. テスト・検証計画 (Verification Plan)

### 3.1 動作検証シナリオ
*   **Scenario 1: New Idea to Project**
    *   `inbox/idea.md` に「RustでTodoアプリ作りたい」と記述。
    *   システムが `library/dev-templates/` を推奨し、`projects/rust-todo-app/` を生成できるか。
*   **Scenario 2: Multi-Agent Collaboration**
    *   生成されたプロジェクト内で、Architectエージェントが設計し、Developerエージェントがコードを書くフローが回るか。

## 4. デプロイ・運用 (Deployment & Operation)
*   **Local Deployment**: ユーザーのローカル環境 (`/workspace`) で完結。
*   **Git Strategy**: `library/` や `config/` はGit管理し、個別の `projects/` はプロジェクトごとの判断とする（またはSubmodule化）。
